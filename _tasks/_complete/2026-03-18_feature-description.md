# Feature: Map-Driven Drawing

**Date:** 2026-03-18
**Status:** Scoped

## Overview

Enable sketches to consume portrait analysis maps generated by the plotter-nodes repo and use them to drive line-drawing algorithms. Maps (density, flow direction, flow speed, importance, coherence, complexity) are loaded from export bundles in `public/maps/`, selected via the UI, and sampled at sketch coordinates to influence particle-based drawing behavior. The first algorithm scatters points by density, traces them along flow fields modulated by complexity, and produces long continuous polylines.

## End-User Capabilities

1. **Drop export bundles into `public/maps/`** and they appear as selectable options in any sketch that uses maps.
2. **Select a map bundle per sketch** via a dropdown in the control panel, with a thumbnail preview of the source image or a representative map.
3. **Toggle a map overlay on the canvas** to visualize the selected map underneath drawn lines, useful for debugging alignment and tuning parameters.
4. **Zoom/scale the sketch output** globally (all sketches) to explore detail at different magnifications without affecting line thickness. This is a new global sketch capability.
5. **Choose cover or fit mode** for how the map is mapped onto the drawing area, preserving the source image's aspect ratio in both cases.
6. **Run a flow-field particle trace algorithm** in the portrait-1 sketch that:
   - Scatters seed points with density weighted by the density map
   - Traces each point along the flow field (flow_x, flow_y)
   - Modulates trace speed by the complexity map (slower in complex regions)
   - Produces long continuous polylines for a configurable number of steps or distance

## Architecture / Scope

### Map Loading Infrastructure

- **Bundle storage:** Export bundles from plotter-nodes are copied into `public/maps/<bundle-name>/`. Each bundle contains `manifest.json`, 7 `.bin` files, and `previews/` PNGs. Vite serves these as static assets.
- **Manifest parsing:** Read `manifest.json` to discover available maps, image dimensions, and metadata. The manifest already contains width, height, map list, and value ranges.
- **Binary loading:** Fetch `.bin` files and decode as `Float32Array`. Maps are loaded on-demand (not all 7 upfront) based on what the sketch requests.
- **Map sampling API:** A `MapBundle` object provides `sample(mapName, x, y)` that:
  - Transforms from sketch cm-coordinates to map pixel-coordinates based on the selected fit mode (cover or fit) and the paper/map aspect ratios.
  - Returns bilinear-interpolated values for smooth sampling between pixels.
  - Returns 0 (or a sensible default) for coordinates outside the mapped region (relevant in "fit" mode where edges may be unmapped).

### Coordinate Mapping

The map's pixel grid and the sketch's cm drawing area have different dimensions and potentially different aspect ratios.

- **Fit mode:** The entire map is visible within the drawing area. Unmapped edges get no map data (default 0). The map is centered.
- **Cover mode:** The map fills the entire drawing area. Some map edges are cropped. The map is centered.
- The sketch author doesn't need to think about pixels — they sample in cm-space and the API handles the transform.

### UI: Bundle Selector

- A Leva-compatible custom parameter type or a standard dropdown listing discovered bundles from `public/maps/`.
- Bundle discovery: scan the maps directory for subdirectories containing `manifest.json` (via a lightweight API endpoint on the Vite dev server, similar to the existing preset API).
- Selecting a bundle loads the manifest and makes maps available. Thumbnail preview displays in or near the control panel.

### UI: Canvas Map Overlay

- A toggleable semi-transparent overlay on the `SketchViewer` canvas showing one of the map's preview PNGs (or a rendered version of the binary data).
- Useful for visually confirming that drawn lines align with map features (eyes, contours, high-density regions).
- Overlay respects the same fit/cover mapping as the sampling API so alignment is exact.

### UI: Global Sketch Scale

- A scale/zoom parameter available to all sketches (part of the framework, not per-sketch params).
- Multiplies all polyline coordinates by the scale factor before rendering. Does not affect stroke width.
- Allows zooming into a region of the drawing to inspect detail or zooming out to see overall composition.
- The SVG export should respect the current scale (exporting the zoomed view).

### First Drawing Algorithm (Portrait-1 Sketch)

The algorithm in `sketches/2026-03-18-portrait-1/`:

1. **Seed scattering:** Generate N candidate points uniformly, then accept/reject based on the density map value at each point (rejection sampling). Higher density → higher acceptance probability. Configurable: total seed count, density influence strength.
2. **Flow tracing:** For each accepted seed point, trace a path by stepping along the flow field (`flow_x`, `flow_y` maps). At each step:
   - Read flow direction at current position
   - Read complexity at current position to modulate step size (slower = shorter steps in complex areas, producing more detail)
   - Advance position by `stepSize * flowDirection * speedFactor`
   - Append position to the polyline
3. **Termination:** Stop tracing when the path exceeds a max number of steps, exits the drawing area, or reaches a max total distance. All configurable via sketch params.
4. **Output:** Return the traced polylines.

Sketch params will include: seed count, step size, max steps, max distance, density influence, complexity influence, speed range, and the standard seed/paperSize/margin.

### Shared Library Code

Reusable pieces that future sketches can import from `src/lib/`:

- **Map loading and sampling** (`src/lib/maps.ts` or similar) — the `MapBundle` class/API.
- **Flow tracing primitive** (`src/lib/flow.ts` or similar) — a function that takes a start point, flow field sampler, and config, returns a polyline. Simple enough to be useful but not over-abstracted.
- **Density-weighted point scattering** — a utility that takes a density sampler and returns scattered points.

Individual sketches compose these primitives with their own logic.

## Technical Details

### Export Bundle Format (from plotter-nodes)

```
<bundle-name>/
  manifest.json          # { version, source, width, height, created, maps: [...] }
  density_target.bin     # float32, [0,1], row-major, little-endian
  flow_x.bin             # float32, [-1,1], unit vector X component
  flow_y.bin             # float32, [-1,1], unit vector Y component
  importance.bin          # float32, [0,1]
  coherence.bin           # float32, [0,1]
  complexity.bin          # float32, [0,1]
  flow_speed.bin          # float32, [0,1]
  previews/              # PNG visualizations of each map
```

Each `.bin` file is `width * height * 4` bytes. Decode with `new Float32Array(arrayBuffer)`. Index a pixel at `(x, y)` with `data[y * width + x]`.

### Bilinear Interpolation

Since sketch coordinates won't land exactly on pixel centers, the sampling API uses bilinear interpolation across the 4 nearest pixels. This prevents aliasing artifacts in the drawn lines, especially in flow field sampling where jagged direction changes would produce visible kinks.

### Bundle Discovery API

A Vite dev server middleware endpoint (like the existing `/__api/presets`) at `/__api/maps` that:
- Lists subdirectories of `public/maps/`
- Returns bundle names and paths to manifest.json and preview images
- Lightweight — just reads the filesystem

## Risks and Considerations

- **Large file sizes:** A 1920×1080 map bundle is ~58 MB (7 maps × 1920 × 1080 × 4 bytes). Loading all maps eagerly could be slow. Mitigation: load maps on-demand as the sketch requests them, and cache loaded maps in memory.
- **Coordinate mapping precision:** The fit/cover transform needs to be exact so the overlay and the sampling agree. Both should share the same mapping function.
- **Performance of particle tracing:** Tracing thousands of particles with per-step map lookups should be fast enough for interactive parameter tweaking. Float32Array access is fast in JS, and bilinear interpolation is just 4 reads + lerps. If performance is an issue, we can reduce particle count or step count interactively.
- **No hot-reload for maps:** Changing the export bundle in `public/maps/` won't trigger HMR. The user would need to refresh or re-select. Acceptable for now.

## Non-Goals / Future Iterations

- **Per-map-channel selection from different bundles** — combining density from one image with flow from another. Interesting but not first pass.
- **plotter-nodes API integration** — uploading an image and getting maps back automatically. Planned for later, and the architecture here (bundle-based loading) will adapt cleanly to API-fetched bundles.
- **Accumulation/density feedback** — comparing desired vs. deposited density and iterating. A near-future experiment building on this foundation.
- **Advanced algorithms** beyond scatter+trace — hatching, stippling, contour-following, etc. These will come as new sketches.
- **Multiple bundles per sketch** — for now, one bundle per sketch instance.
- **Map generation within this repo** — maps are always generated externally by plotter-nodes.

## Success Criteria

1. A map bundle copied to `public/maps/` is discoverable and selectable in the UI.
2. Selecting a bundle shows a preview thumbnail in the control panel.
3. The canvas overlay toggles on/off and visually aligns with the drawn output.
4. The portrait-1 sketch produces visible, aesthetically meaningful lines that clearly respond to the underlying maps (denser where the density map is high, flowing along the flow field, varying detail with complexity).
5. Changing sketch parameters (seed count, step size, etc.) produces meaningfully different results in real-time.
6. The global scale/zoom works on all sketches and SVG export respects it.
7. Cover and fit modes both work correctly with no stretching/distortion.
